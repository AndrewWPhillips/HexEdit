Other Tools

 

The following tools perform operations on the current selection.  All these tools can handle selections of any size (ie 4GBytes or more).  If the operation is to take a long time then progress is displayed in the status bar.  You can also abort these commands at any time by pressing Esc (or any) key.

 

Conversions

 

The Convert sub-menu on the Operations Menu provides commands to convert between different character sets.  You can convert between ASCII and EBCDIC and between the ASCII extensions of ANSI and IBM/OEM.  To convert just select the bytes you want to convert and use the conversion command.

 

For all conversions the length of the selection does not change.  Any characters that cannot be converted are changed to null (ie zero) bytes.

 

If the current selection is very large (ie in the hundreds of MByte or GByte range) then HexEdit may need to save the result in a temporary file to avoid memory exhaustion.  You must ensure you have enough space in the area that Windows creates temporary files (usually C:\Documents and Settings\<login name>\Local Settings\Temp) if you convert very large files.

 

Checksums, CRCs and MD5

 

The Checksum sub-menu on the Operations Menu provides commands to calculate checksums of the current selection.  These commands leave the result of the operation in the calculator.

 

The checksum commands provide 1, 2, 4 and 8 byte checksums of the selection.  This is simply done by adding all bytes, words, double-words or quad-words of the selection together (throwing away any carry bits).  These commands are faster and more convenient than creating a keystroke macro to generate the checksum.  However, for other types of checksums (eg, XOR checksums) you will need to use the calculator and a keystroke macro.

 

The checksum commands require that the current selection length is a multiple of the operand size.  For example, for a 32-bit checksum the selection length must be a multiple of 4 bytes.  You can easily check this by holding down the shift key, which not only tells you the selection length but the largest binary divisor of the length of the selection in square brackets.  For a 32-bit checksum the number in the square brackets must be 4, 8, 16 etc (ie, not 1 or 2).

 

The CRC commands provide better error detection than simple checksums.  CRC32 provides a 32 bit checksum which is better than CRC-CCITT which is a 16 bit checksum.  These commands can handle selections of any length.  Note that the CRC-CCITT provides the correct standard implementation of this algorithm -- many other programs calculate the incorrect value (due to some incorrect source code being published on the Internet).

 

Finally, the new MD5 message digest command (also called a cryptographic checksum or one-way hash function) provides a value that ensures a file has not been changed in any way.  While a CRC is excellent for detecting random errors and noise it will not protect against deliberate tampering - that is it is fairly easy to generate data that has a known CRC.  But (at the present time and in the foreseeable future) it is computationally infeasible to do the same for an MD5 value.  Note that the MD5 algorithm generates a 128-bit result - anything less and it would be susceptible to brute force attacks.

 

MD5 values are quite commonly used now to verify that publicly distributed files have not been modified for nefarious purposes.  For example, you can obtain the MD5 value for the HexEdit installation files from the HexEdit download page (http://hexedit.com/download.htm), but obviously you shouldn't use the downloaded version of HexEdit to check its own installation package, but using the previous version of HexEdit (eg using HexEdit 3.1 to verify the MD5 value for HexEdit 3.2) should be OK as long as you don't suspect your currently installed copy.

 

Export and Import

 

The Export and Import sub-menus on the File menu provide commands for writing binary data as a text file and reading such files.  These types of files are often used with embedded systems test environments, EPROM programmers etc, but can also be useful for sending binary data over links the don't allow all binary values.

 

The Motorola-S format supports three types of records (S1, S2, and S3) which support 16, 24 and 32 bit addresses.  HexEdit supports the import and export of all three record types (unlike many other hex editors).  One or more of these export options may be disabled if the length of the selection combined with the starting address means that the address size is too small to be represented in a particular record type.  The S5 record count is checked to ensure whole lines of data are not lost (or a warning is shown if the S5 record is not present).

 

HexEdit exports standard Intel-hex files and imports standard as well as non-standard (but commonly generated) files.  All record types are handled and checked for errors but only "00" and "01" records types are required.

 

HexEdit performs full error checking when importing Motorola-S and Intel-hex files including checksums (unlike some hex editors which ignore errors).  Further, all records must be contiguous so that you do not inadvertently move data to the wrong location.

 

When exporting you have the option of specifying the starting address in the output records using Export section of the System Options page, or you can use the address of the selection within the file being edited.  (All exported records are contiguous.)  You can also specify how many bytes of binary data are put into each output record using the "Bytes per line" option.

 

The "hex text" options provide no error checking but simply works with files that consist of pairs of hex digits each representing a byte.  When importing hex text HexEdit reads pairs of hex digits and converts them to the corresponding byte value; a single hex digit followed by whitespace becomes a single byte with the high nybble set to zero.  When exporting hex text the "Bytes per line" option of the System Options page is used but the starting address options are ignored.

 

Comparisons

 

You can compare the data in any two (non-minimised) windows using Alt+C, even two parts of the same file.  The comparison starts at the first byte of the selection in both windows.

 

If a difference is found the offending bytes are selected in both windows – pressing Alt+C again will continue the comparison at the following byte. It’s often useful to tile (vertically) the two windows being compared and turn on Autofit mode before starting the comparison to easily see the results.  Also if you turn on the highlighter all differences are automatically highlighted as you find them.

 

While a long comparison (or search) is in progress the Address Tools in the Edit Bar are updated with the current address to show progress. If no differences are found (ie, the files are the same in both windows up to the end of file) then a message to that effect is displayed in the status bar and the cursor is not moved. If a difference is found (ie, a different byte was encountered or end of file was encountered in but one of the files) then a message is displayed in the status bar indicating the number of bytes that were the same before the difference was found.
