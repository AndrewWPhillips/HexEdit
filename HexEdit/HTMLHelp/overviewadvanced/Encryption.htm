Encryption

 

HexEdit Pro allows you to encrypt and decrypt all or part of a file by selecting the bytes and using the encryption or decryption command.  It provides all the stream and block encryption algorithms available under the Windows "Crypto API" as well as a good internal encryption algorithm.

 

The Crypto API and CSPs

 

The Crypto API was introduced with Windows NT 4.0, mainly to address security issues to do with the growing use of the Internet.  This API does not itself do anything but simply provides a standard way for Windows programs to talk with Cryptographic Service Providers (CSPs).  CSPs are DLLs provided by third party software providers (although Microsoft provides the "Microsoft Base Cryptographic Provider v1.0" CSP).

 

Depending on the software you have installed on your computer, you will typically have a few CSPs, perhaps many, installed.  Each CSP provides one or more encryption algorithms or ciphers.  Different CSPs may provide different implementations of the same algorithms, but, in general, you should always decrypt using the same CSP and algorithm that you encrypted with.  If you are encrypting on one machine and decrypting on another you must ensure the same CSP/algorithm is available on both machines.

 

If you are using Windows 95, and HexEdit Pro only shows one encryption algorithm (the HexEdit Pro internal algorithm) then the Crypto API is not available on your computer.

 

Algorithms

 

HexEdit Pro only supports use of the single-key or symmetric ciphers.  (This is the traditional type of computer encryption where the same password or key is used to encrypt and decrypt.)  HexEdit Pro does not support public-key ciphers, as these are only provided in the Crypto API for encryption of session keys.  The Crypto API provides for two types of single-key cipher: block and stream.

 

Block encryption works by encrypting all the bits of a fixed size block at once.  If the number of bytes to be encrypted is not a multiple of the block size then pad bytes will need to be added, hence block ciphers provided by the Crypto API increase the numbers of bytes when encrypting by up to 2 blocks.  Hence, when encrypting in HexEdit Pro with a Crypto API block cipher the length of the selected bytes will increase after encryption, and decrease back to the original size after decryption.  (This is not true for the HexEdit Pro internal algorithm - see below.)  The Microsoft Base Cryptographic Provider v1.0 provides the RC2 block encryption algorithm.

 

Stream encryption, as its name suggests can take an unencrypted stream of data and produce an encrypted stream.  Hence, unlike block encryption, you can encrypt any number of bytes with stream encryption.  In HexEdit Pro, when you encrypt with a stream cipher, the encrypted length is the same as before encryption.  The disadvantage of stream encryption is that the algorithms are typically not as secure, and implementations are slower than block encryption.  The Microsoft Base Cryptographic Provider v1.0 provides the RC4 stream cipher.

 

The HexEdit Pro internal algorithm is a block cipher with a block size of 8 bytes (64 bits).  As it does not provide padding of any incomplete last block it does not need to increase the length of the encrypted data.  However, to encrypt or decrypt using this algorithm the length of the current selection must be a multiple of 8 bytes.

 

Passwords

 

The password is the "key" that is used to encrypt and decrypt.  It does not matter how good your encryption algorithm is if you choose an easily guessed password.  Hence a good password is at least 8 characters long, is not a common word or name in a common language, and should include some non-alphabetic characters.

 

HexEdit Pro has a minimum password length which defaults to 8.  However, as encryption requirements vary this can be changed.  For example, if you are only using HexEdit Pro for testing Crypto API functions then you might want to reduce the minimum required password length.

 

You can also mask your passwords using the "Mask password" check box.  This is useful if anyone could possibly view your screen while you are entering the password.  However, since you have no feedback about what you have typed, you have to enter a masked password twice to catch typos, otherwise you may not be able to decrypt your valuable data again.

 

Particular care must be taken with passwords and keystroke macros.  If a password is entered while recording a macro, and the macro saved to file then the password is stored in the file for all to see.  Typically, you would not enter a password while recording a macro, and to ensure that the user is prompted for a password when encrypting or decrypting you can record the "Clear Password" command in the macro; if no password has been entered since last cleared when the Encrypt or Decrypt command is invoked then HexEdit Pro will prompt for one.

 

Public Key Encryption

 

Another form of encryption that HexEdit Pro does not (yet) support is public key encryption. This has many excellent uses that I will not go into here.  In this form of encryption the two keys (encryption/decryption) are not the same.  One key (the public key) can be easily deduced from the other key (the private key) but it is computationally infeasible to deduce the private key from the public key.  A common public key encryption algorithm is RSA.

 

Unfortunately, known public key algorithms are very slow compared to single-key algorithms.  However, it is not necessary to encrypt all the data using public key encryption.  The data itself can be encrypted using a single-key algorithm using a key that is itself encrypted using a public-key algorithm.  This system of using two algorithms (a public key one and a single-key one) effectively gives the advantages of public key encryption at close to the speed of single-key encryption.

 

To do this the single-key encryption is performed using a randomly picked number for the encryption key.  For example, to encrypt and send data, a random number is generated and encrypted using the public key encryption algorithm using the receiver's public key, and sent.  Then the data itself is encrypted with the single-key algorithm using this random key.  As the random value is generated and only used for only a short time, or session, it is called a session key.

 

The Crypto API uses public key encryption in this manner.  That is public key encryption is only used to encrypt session keys for single-key encryption algorithms.
