Decompression (ZLIB) command

 

Decompresses the current selection using the ZLIB (de)compression algorithm.  The selection must have been previously compressed using ZLIB (and smaller or the same window size - see below) for the decompression to succeed.  Since decompression will usually increase the length of the selected bytes you are asked if you want to switch to INS mode if currently in OVR mode.

 

If the compressed data has been corrupted (or it was never compressed with ZLIB in the first place) then you will get the error message "The compression data is corrupted".  In this case you are given the option of continuing or aborting.  If the compression data was written with data recovery (flush) points then HexEdit may be able to recover some of the data despite the corruption.

 

The decompression may also fail depending on some settings in the Compression Settings Dialog:

  • if the current "window size" is smaller than the window size that was used to compress the data.

  • if a (zlib or gzip) header was used during compression but no header (raw) is used for decompression

  • if a no header (raw) was used during compression but a (zlib or gzip) header is used for decompression

Note: Apart from the window size and whether or not a header is used no other settings from the dialog are used for decompression (but all settings are used for compression of course).  The compressed stream provides information to the decompression algorithm that allows it to adapt -- for example, either type of header (zlib or gzip) can be handled.
