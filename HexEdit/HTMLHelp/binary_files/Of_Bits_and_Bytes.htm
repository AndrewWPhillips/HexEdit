Representations

 

When we write the bits of a byte as 8 bits each having the value of 0 or 1, we are actually writing the numeric value of the byte as a binary number.  This number can represent a numeric value or it can just be used to identify one of many possible things.  For example, a byte can be used to represent a character (alphabetic, digits, punctuation etc), where it being a number is not particularly relevant, except, perhaps, for defining an order to the characters.  We will talk about character sets in a later section.

 

Hexadecimal Notation

 

A group of X bits can store 2^X different values. So a single bit can store 2^1 or just 2 values (which we will call 0 and 1). A byte has 8 bits so it can store 2^8 or 256 different values. Similarly a nybble can store 2^4 or 16 values, a word 2^16 or 65,536, a double word 2^32 or 4,294,967,296, and a quad word 2^64 or a lot.

 

The number of different symbols (or digits) used to display numbers is called the radix or base.  Decimal numbers use 10 different digits, binary numbers use 2, while hexadecimal numbers use 16.  Generally, people are used to seeing and manipulating numbers using the decimal or base 10 system, not because it is the best system, but for historical (and anatomical) reasons.  In a decimal system there are 10 different digits (0, 1, … 9). It turns out that the easiest way for electronic circuits to store and manipulate numbers is using the binary or base 2 system.  In this system there are just 2 digits (0 and 1).

 

When people need to work with binary data, looking at large numbers of zeroes and ones can be tedious and error-prone.  This is where hexadecimal or base 16 is useful.  But for hexadecimal (or hex for short) we need 16 digits. So to the normal 10 decimal digits (0 to 9) we add the 6 alphabetic characters A to F which represent the values 10 to 15.

 

Why not display binary data as decimal rather than hex, you may ask?  Well it turns out that mathematically, binary and hex are closely related, since 16 is a power of 2 (ie, 16 = 2*2*2*2).  Any hex digit can represent exactly 4 binary digits (ie., one nybble). Hence it is simple to convert between binary and hex.  This is important as binary values are not always used to store numbers - they may be used to store a sequence of flags, for example.  It is relatively easy to convert hex digits to binary to see which flags are on; it would be more difficult to convert from decimal back to binary.

 

Well then why not use octal (base 8), you may ask?  Trust me, hex is better.  Octal was popular for a time, but hex has the advantage that a byte can be displayed as 2 hex digits. Note that since many hex numbers (especially if they happen to not contain any of the digits A to F) may be mistaken for innocent decimal numbers we need to distinguish them from decimal to avoid confusion. There are several conventions. A very old convention is to precede the number with a dollar sign ($). A convention that comes from Intel assembler is to append a lower case ‘h’. While C/C++ programmers use a ‘0x’ prefix. Another convention is to simply prepend or append an 'x'.  So the number 16 (decimal) may be written in hex as $10, 10h, 10x, or 0x10.

 

Base Conversion

 

You may remember from school about how numbers are stored in different bases. In general a number, say ABCD in base X is A*X*X*X + B*X*X + C*X + D. That is, the number is the sum of the values for each digit. The value for digit N is the digit times X^N (where N = 0, 1, 2, …). Don’t forget that anything to the power zero (X^0) is one.

 

For example, if you convert the hex number 468A to decimal you get 4*16^3 + 6*16^2 + 8*16^1 + 10*16^0 = 4*4096 + 6*256 + 8*16 + 10*1 = 18,058 (base 10). Converting the same number to binary is easier since each hex digit converts to 4 binary digits. So 468A = 0100 0110 1000 1010 (base 2).

 

 

The point of all this is that files are stored as a sequence of bytes, and bytes are most easily shown in hex (because, as we saw above, it’s too hard to convert decimal to binary, octal is ugly and binary would just take up too much screen space). This is the reason that HexEdit’s (main) display of binary files is a sequence of pairs of hex digits representing the bytes of the file.
